## 1. Design Rationale

OpenSeal is built on the principle that **"Execution is Proof."** In the era of AI agents and automated infrastructure, verifying API response integrity without relying on a centralized authority is the critical link for establishing total trust.

### Why OpenSeal?
- **Universal Integrity**: Unlike TEEs that require specialized hardware or ZK proofs with high computational overhead, OpenSeal provides a **Practical Integrity Layer** that works across any language and OS.
- **Protocol Over Infrastructure**: OpenSeal is not just a runtime; it is a **Standard Protocol** that binds a result to a specific, immutable code identity.
- **Strategic Moat**: By focusing on the logic-to-result binding, OpenSeal creates a standard that is easy to adopt but difficult to replicate in its operational security and ecosystem integration.

## 2. Threat Model & Guarantees

We define the boundaries of what OpenSeal explicitly protects to ensure an "Honest Execution" model.

### âœ… What We Guarantee
- **Logic Integrity**: Verifiers can be 100% certain that the result was generated by the *exact* same source code (A-hash) publicly audited or registered.
- **Freshness (Anti-Replay)**: Every response is bound to a specific, one-time challenge (Wax), preventing attackers from reusing old, valid responses.
- **Sandbox Isolation**: The application runs in a sanitized environment where the parent process (OpenSeal) intercepts all I/O to prevent silent manipulation.

### âŒ What is Out of Scope
- **Host Memory Security**: We do not claim to prevent a super-user with physical access from hooking memory *during* execution. This is the domain of TEEs.
- **External Data Truth**: OpenSeal proves the **Logic was honest**, not that the external data fetched by that logic (e.g., an external exchange rate) is objectively true.

## 5. Privacy & Integrity Commitment

The OpenSeal Protected Runtime is built on a **Zero-Backend Policy**:
- **Zero Data Collection**: No user data, execution results, or metadata are collected or stored.
- **Zero Outbound Communication**: The core generation engine operates in complete isolation, performing no unauthorized network calls.
- **Auditable Boundary**: The interface boundary (ABI/IPC) between the open CLI and protected runtime is clearly defined and auditable.
